\# ADR 006 — LLM Pulse Digest



\## Decision

Add a periodic narrative summary of pipeline state ("Pulse Digest"),

generated by calling the Anthropic API, with a deterministic rule-based

fallback when no API key is configured or the call fails.



\## Why This Matters

Every other feature in this dashboard makes the person read a number and

infer what it means: "Sentiment Index: -1" requires the reader to know

what -1 means relative to the scale. A digest inverts that — instead of

"go read six charts to find out what's happening," it says what's

happening, in one sentence, up front. This is the one feature in the

whole project explicitly aimed at \*understanding\*, not just \*display\*.



\## Implementation

\- New process: `processor/digest\_generator.py` — same pattern as

&#x20; `spark\_processor.py` and `stream\_join.py`: a long-lived loop, polls on

&#x20; an interval, writes a CSV, the dashboard just reads the CSV. No new

&#x20; architecture, just another producer of a CSV the dashboard already

&#x20; knows how to render.

\- Input stats pulled from every existing CSV output (`data\_posts.csv`,

&#x20; `data\_volume.csv`, `data\_alerts.csv`, `data\_correlation.csv`,

&#x20; `data\_drift.csv`) — total posts, velocity, sentiment mix, top posts by

&#x20; engagement, active anomalies, new-vs-top correlation, and the

&#x20; highest-scoring drift event. The digest is a summary of everything else

&#x20; the pipeline already computes, not a new signal.

\- Model: `claude-haiku-4-5-20251001` — fast and cheap, appropriate for a

&#x20; short structured summarization task run every few minutes, not a

&#x20; reasoning-heavy task that would justify a larger model.

\- Refresh interval: 300s default (`DIGEST\_INTERVAL\_SECONDS`), independent

&#x20; of the dashboard's own refresh rate, so API cost stays predictable

&#x20; regardless of how often someone's staring at the browser tab.



\## Resilience (the actual hard part)

An LLM call is an external dependency the rest of the pipeline doesn't

have, so it gets the same treatment as the HN API calls in

`reddit\_producer.py`:

\- \*\*No hard failure without a key.\*\* If `ANTHROPIC\_API\_KEY` isn't set,

&#x20; the generator produces a deterministic, template-based digest from the

&#x20; same stats instead of crashing or producing nothing. A demo, a CI run,

&#x20; or a reviewer without an API key still sees a real digest.

\- \*\*Fail fast on auth errors, retry on transient ones.\*\* A bad/missing

&#x20; key won't be fixed by retrying, so `AuthenticationError` short-circuits

&#x20; straight to the fallback. Genuinely transient failures (timeouts,

&#x20; 5xx, rate limits) get 3 attempts with exponential backoff (2s, 4s).

\- \*\*Graceful shutdown.\*\* SIGINT/SIGTERM finish the current cycle and

&#x20; exit cleanly, consistent with the other long-lived processes in this

&#x20; pipeline.



\## What We Found

Testing the fallback path is what actually caught the design flaw worth

writing down: an early version retried 3 times (with backoff) even on a

401 — meaning a missing API key cost \~7.5 seconds of pointless waiting

per cycle before falling back. Since auth errors can't be fixed by

retrying, that's pure latency with no benefit. Splitting "retry-worthy"

errors from "fail-fast" errors dropped that to under a second.



\## Trade-offs

\- The rule-based fallback is noticeably more mechanical than the LLM

&#x20; version — it's accurate, not eloquent. That's an intentional trade:

&#x20; a working degraded mode beats a missing feature.

\- Digest quality is bounded by how good the underlying stats are. If

&#x20; `gather\_stats()` is wrong, the LLM will confidently narrate the wrong

&#x20; thing — the digest doesn't independently verify anything, it just

&#x20; narrates what it's given.
